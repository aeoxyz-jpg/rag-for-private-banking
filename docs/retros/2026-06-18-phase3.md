# Retro — Phase 3 (semantic-layer value/storage, reranking, GraphRAG triage)

_2026-06-18. Four sub-projects: 3a (semx), 3d (rerankx), 3b (semrepr) built & merged; 3c (GraphRAG)
deferred on evidence. All standalone harnesses mirroring kgx/semx convention; production layer untouched._

## One line

Validated the semantic layer's governance value (3a) and storage representation (3b), tested a reranker
over pillar A (3d), and **triaged GraphRAG out with a precondition probe** (3c) instead of building a
circular experiment. The throughline: every verdict pre-registered, every "fancier component" measured
against the simpler baseline, every pivot honestly relabeled.

## Technical lessons (this codebase)

### 3a — semx (B text-to-SQL vs E semantic layer)
- **The pre-registered 0.2 drift threshold sits at B's central tendency.** Single run → B drift 0.208
  (JUSTIFIED); re-run → 0.173 (INCONCLUSIVE); 7 runs → **0.197 ± 0.013**, verdict splits 4/3 on noise.
  The magnitude (~20% drift, concentrated in composite KPIs `churn_risk` 0.29–0.50, `days_since_contact`
  0.25) is the result; the binary label is not. `rm_assistant/semx/report.py:verdict` thresholds.
- **E doesn't cleanly abstain — it mis-routes ~78% of ungoverned probes** (picks a nearby governed metric,
  silently answers the wrong one). The first run reported `E_abstain_rate=0.30` which *understated* this;
  the fix was to record `chosen_metric` on probe runs (`semx/runner.py:_score` probe branch) and split
  abstain vs mis-route in `semx/report.py:coverage`. Lesson: a "coverage" number can hide a worse failure
  mode — capture *what* the engine did, not just pass/fail.
- Canonical gold comes from `ontology.metric_sql(name)` executed once — never from either engine's output.
- Stability harness: `scripts/run_semx_stability.py` (N independent runs → `docs/semantic_stability.md`).

### 3d — rerankx (RRF vs RRF+reranker)
- **Ollama 0.30.8 has NO native rerank endpoint** (`/api/rerank` → 404; PR ollama/ollama#7219 unmerged).
  `dengcao/Qwen3-Reranker-0.6B:Q8_0` *loads for generation but emits garbage* (commas for any input — even
  "is the sky blue? yes/no"); `pdurugyan/qwen3-reranker-0.6b-q8_0` is **embedding-only** ("does not support
  generate"). Verify a model actually works on *this* Ollama before designing around it.
- **Pivot: LLM-as-reranker** (`qwen2.5:3b-instruct`, P(yes) from yes/no logprobs via `/v1/chat/completions`,
  which *does* expose `top_logprobs` on 0.30.8). Qwen3-Reranker's official "Judge whether the Document meets
  the requirements… yes/no" prompt is **too strict for a general LLM** (answers "no" to a clearly relevant
  doc); a plain relevance prompt ("Is this document relevant to the question? yes/no") discriminates cleanly
  (rel ~1.0 / irrel ~0.0). `rm_assistant/rerankx/reranker.py:_SYSTEM/_user`.
- **The reranker NET-HURT** ranking (MRR 0.62→0.51, recall@1 0.47→0.39, +8.4s/q). Mechanism visible in the
  actionable-vs-full split: it promotes buried golds (actionable MRR 0.38→0.58) but reshuffles RRF's
  already-correct rank-1s downward. Pool recall 0.86 → 14% are first-stage misses no reranker can fix.
- `rm_assistant/retrieval/vector_pillar.py:fused_candidates` exposes the RRF pool (refactor: `search` reuses
  it). Hard set = each question pinned to one note of a multi-note client → siblings are distractors
  (`scripts/gen_rerank_hardset.py`, `rm_assistant/eval/gold/rerank_hard.json`).

### 3b — semrepr (storage representation)
- **A declarative dbt/Cube-style spec compiles to SQL that reproduces every canonical number** (validated
  by a planning-stage spike before the plan was written). `share_of_wallet` becomes a **native ratio
  reference** to `aum` (vs the baseline's `{{ref}}` string-substitution); `churn_risk`'s clamped weighted
  sum still needs a **raw-SQL escape hatch** (1/2 composites native). `rm_assistant/semrepr/dbt_repr.py`.
- **RDF (`rdflib`, pure-Python) models composition as `dependsOn` lineage but `assemble_sql` is the same
  string-substitution as the baseline** — the SQL body stays a literal, so RDF *generates* nothing (0/2
  native by the typed-generation criterion → verdict `keep`). Honest distinction: lineage ≠ SQL generation.
- Equivalence compared at **6 decimals** (`semrepr/equivalence.py:_result`) — float re-assembly differs in
  the last bits. Verdict `migrate` iff equivalence + ≥1 native composite + queryable + ≤1.5× verbosity
  (`semrepr/report.py`). Result: **dbt-style → migrate, RDF → keep**.
- Verbosity comparison (baseline re-dumped YAML vs raw source files) is **conservative against migrate** —
  it counts the alternatives' comment/label lines, so the bias can't flip a verdict (final reviewer checked).

### 3c — GraphRAG, deferred
- **Precondition probe before building.** `scripts/probe_corpus_entity_structure.py`: of 1623 single-client
  notes, **0% mention a company, 1.3% mention another client** → near-zero cross-document entity structure.
  GraphRAG (extract entities → link across docs → traverse/aggregate) has nothing to exploit; the
  relationships that exist (household/employer/UBO) are already structured (Phase 2 covered them). Building
  here + authoring matching questions would repeat the §6 Q5 circular-gate. Global sensemaking (where
  GraphRAG would win) has no canonical gold. → deferred to a Phase-4 purpose-built-corpus project.

## Pitfalls / verification methods
- **Spike the crux during planning.** 3b's compiler equivalence was proven on the real warehouse *before*
  writing the plan → implementers hit zero surprises. 3d did not spike model availability → the broken-GGUF
  wall hit mid-execution (caught only because the plan had a real-Ollama smoke).
- **Build a real-integration smoke into the plan, with "fail → report, don't redesign."** rerankx Task-2's
  smoke caught the garbage-output GGUF immediately and forced a clean pivot instead of a fudge.
- **Multi-run before trusting a binary verdict** when the threshold is anywhere near the measured value.
- Each experiment writes records to gitignored `data/*.json` + a committed `docs/*_eval.md`.
